# UX改善・パフォーマンス向上実装計画

## 概要
アプリケーションの各所で見られる「もっさり感」を解消し、ユーザーがストレスなく操作できるように UX を改善します。主に Next.js の機能を活用したローディング表示の追加と、フォーム送信時の即時フィードバックの実装を行います。

---

## 1. ページ遷移の改善 (Page Transitions)
Next.js のストリーミング機能を活用し、ページ遷移中に「読み込み中」であることを視覚的に伝えます。

### 実装内容
- **[NEW]** `app/loading.tsx`
  - アプリ全体の共通ローディング画面。中央にモダンなスピナーとロゴを表示します。
- **[NEW]** `app/app/loading.tsx` (必要に応じて)
  - 認証後のメインエリア専用のローディング。サイドバーを残したままコンテンツエリアのみローディングを表示します。

---

## 2. 認証・フォーム送信の改善 (Form Feedback)
ボタンをクリックした直後に状態を変化させ、二重送信の防止と処理中であることの視覚的ヒントを提供します。

### 実装内容
- **[MOD]** `app/login/page.tsx`
  - ログイン・サインアップボタンを別コンポーネント化し、`useFormStatus` を使用してローディングスピナーを表示するように変更します。
- **[MOD]** `components/transactions/TransactionForm.tsx`
  - 送信中（`isSubmitting`）の際に、ボタンのテキスト変更やオーバーレイ表示による視覚的フィードバックを強化します（現在はボタン内スピナーのみ）。
- **[MOD]** 各種設定画面 (`AccountList`, `CategoryList` 等)
  - 削除や追加ボタンにローディング状態を追加します。

---

## 3. データベーススキーマの拡張 ([NEW])
「場所(大分類)」を保存するためのカラムを `transactions` テーブルに追加します。

### 実装内容
- **[SQL]** `ALTER TABLE public.transactions ADD COLUMN IF NOT EXISTS place_category TEXT;`
- 分類内容: `net` (ネット), `convenience` (コンビニ), `supermarket` (スーパー・小売), `restaurant` (飲食店), `vending` (自動販売機), `other` (その他)

---

## 0. 準備: 既存データのクリーニング ([UPDATE])
重複と残高の不整合を解消するため、現在の `transactions` データを一度すべて削除し、正規化されたロジックで再インポートします。
また、自動作成された口座のうち、名称が不適切なもの（「Olive支払い」など）も整理対象とします。

---

## 4. データ移行の詳細 (Data Migration)
指定されたフルデータ CSV ファイルからインポートを行います。

- **インポート元**: `docs/収支表 - フォームの回答 1.csv`
- **対象ユーザー**: `c708c711-42e2-4692-9104-38dcaeb96fb3`

### CSV 列構成
0: タイムスタンプ, 1: 日付, 2: 収支, 3: (空), 4: 金額, 5: ジャンル, 6: 支払い方法・移動元, 7: 場所(大分類), 9: 内容, 11: 移動先・収入先, 13: カード名, 14: メモ(店舗名)

### マッピングロジック
- **日付 (`date`)**: 1列目 (yyyy/mm/dd -> yyyy-mm-dd)
- **種別 (`type`)**: 2列目 (支出→expense, 収入→income, 移動→transfer)
- **金額 (`amount`)**: 4列目
- **カテゴリ (`categoryId`)**: 5列目 (ジャンル) を元に検索
- **支払い元 (`fromAccountId`)**: 
  - 収支が「支出」かつ 6列目が「クレジットカード」の場合 → 13列目 (カード名)
  - それ以外 → 6列目
- **受け取り先 (`toAccountId`)**: 11列目 (移動先・収入先)
- **内容 (`description`)**: 9列目 (内容)
- **場所 (`place`)**: 14列目 (メモ) ※14列目が空なら 7列目 (場所)
- **場所(大分類) (`placeCategory`)**: 7列目 (場所(大分類))
  - `ネット` -> `net`
  - `コンビニ` -> `convenience`
  - `スーパー・小売` -> `supermarket`
  - `飲食店` -> `restaurant`
  - `自動販売機` -> `vending`
  - `その他` -> `other`

## 5. 重複排除と正規化のロジック ([NEW])
「重複消費」を防ぎ、資産残高を正確にするため、以下の高度なパースロジックを導入します。

### a. アカウント名の正規化
- **「支払い」表記の統一**: `Olive支払い` や `デビュープラス支払い` は、末尾の「支払い」を削除して `Olive`, `デビュープラス` として扱います。これにより、銀行からの「移動（振替）」がカード口座のマイナス残高を正しく相殺するようになります。
- **カード名の優先適用**: 収支種別に関わらず（「支出」だけでなく「移動」も含め）、支払い元が「クレジットカード」で「カード名」列が入力されている場合は、カード名のアカウントを優先的に使用します。

### b. 同一取引の検知とマージ
- **厳密な一致判定**: 「日付」「金額」「内容」が完全に一致するレコードが複数存在する場合、以下の優先順位で1件のみを採用します：
  1. 口座名が具体的（例: `NL`, `楽天カード`）なもの
  2. 口座名が抽象的（例: `クレジットカード`）なもの
- これにより、スプレッドシート上の複数シートから重複してエクスポートされたデータを除外します。

- インポート後の資産残高が、スプレッドシート上の「総資産合計」と理論的に一致するか確認。
- 支払い（振替）によってカード口座の残高が増加し、支払いが完了していることを確認。
